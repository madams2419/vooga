VOOGASalad : Analysis

This part is an individual effort (like an exam) and should represent your own thinking about the project.

Write one last piece of code to demonstrate what you have learned during the semester and reflect on your project experience.

Submission

Create three Pull Requests on GitHub in three separate repositories to represent your complete analysis.

  • Written analysis: create a branch, within your analysis repository, called VOOGASalad. Write your analysis in a single file, ANALYSIS.md, in this new branch then create a Pull Request from this branch back to master. To make it possible for the course staff to read and give you individual feedback, it must be completed using GitHub's markdown format (here is a web-based WYSIWYG editor for markdown).
  • Code masterpiece: create a branch, within your voogasalad_XX repository, called analysis_NETID. Refactor one or more classes from your project in this new branch then create a Pull Request from this branch back to master. These changes do not need to be compatible with the current code in the master branch (i.e., you can change the signature of an existing method if you think it improves the overall code).
  • SLogo addition: create a branch, within your slogo_teamXX repository, called addition_NETID. Add or update one or more classes in this new branch then create a Pull Request from this branch back to master. These changes must be compatible with the current code in the master branch (i.e., this Pull Request should be able to be accepted and work with the existing stable version of the code).

Each class that you want to be considered as part of your masterpiece must compile and have exactly this comment at the top:

// This entire file is part of my masterpiece.
// YOUR NAME

Specifications

Your analysis should be divided into three parts:

  • project journal (~1000-1500 words)
  • review of the project's design (~2500-3500 words)
  • small, but functionally significant, code submission that shows off how well you can write code (~300-600 words and ~100-200 lines of code)

This analysis is your opportunity to demonstrate your knowledge of design concepts represented (or not) in the project's code. If it is also an honest and thoughtful reflection, it can help you identify specific actions you can do to improve future projects.

The following sections must be included in your analysis. Questions are provided to get you started thinking about what to consider within each section; you may, of course, include anything else you feel is relevant. In other words, do not be limited by these questions, but respond to them at least.

Project Journal

This section represents your personal view of the project's planning and progress. Analyze your own role and the roles each person assumed in the team (not their personalities) and how the outcome could be improved. Describe specific, important, events during the project's time line to justify your comments rather than general terms like "always late", "writes bad code", etc.

  • Time Review
    Reflect on how you spent your time while working on this project.
    • Report how many hours you spent on the project as well as its start and finish dates (do not forget to include time spent planning and in meetings).
    • Describe how you spent your time (i.e., coding new features, refactoring, testing, reading, designing, debugging, documenting, or meeting with team members).
    • Describe how you personally managed your code (i.e., develop, test, package, and push it)?
    • Reflect on what what tasks were easiest for you to do and what were hardest; what were good uses of your time and what were bad.
  • Teamwork
    Reflect on how the team worked together, focus on each person's role in contributing to the team.
    • Report how many hours the team spent planning and designing the project and when these design meetings occurred.
    • Describe each person's role in the team (including outside people with whom you collaborated) and for what parts each person was responsible.
    • How did you feel about the communication within the team (was it too much, too little, just right) and why? Were there any warning signs of problems that occurred later in the project?
    • Describe the team's plan for completing the project and how stable the plan was (from choosing roles to how components would fit together to avoiding deadlocking to how to integrate, update, and test the code).
    • How did the plan and team roles hold up to the project's extensions to the original specifications?
  • Issues
    Reflect on how you worked on the project's issues.
    • How did you manage your code individually (i.e., develop, test, package, and push it)?
    • Look back over your commit messages, do you feel they accurately represent your contribution to the project?
    • Report how many issues you took responsibility for, closed, and worked on but did not close.
    • Choose one issue you closed and describe its progress to completion in detail:
      • what design choices did you face in working on this issue?
      • how did you work with others on the team to determine the scope and goals of this issue?
      • what, if any, work did you need to do on this issue after it was initially closed?
  • Conclusions
    Reflect on how effectively your team worked together and how much you contributed to that effectiveness.
    • Did you over- or underestimate the size of this project? How can you estimate better in the future?
    • Did you take on enough responsibility within the team? Did you keep your team informed about your progress?
    • Which parts of your code required the most editing? why?
    • To be a better designer, what should you start doing, keep doing, or stop doing?
    • To be a better teammate, what should you start doing, keep doing, or stop doing?
    • If you could work on one part right now to improve any aspect of your project, what would it be?
Design Review

This section represents your personal view of how your program is organized and why it was done that way. It can be a chance for you to describe how you might fix a problem or refactor the design if you did not agree with the team or did not have time to actually change the code. You might also learn something by looking at someone else's code.

If you think something is good, justify your opinion. If you think something is bad, justify your opinion and suggest how to improve it. Use specific code examples and reasons related to principles or patterns discussed in class or in the readings rather than general terms like "clearly/obviously", "good/sucks", "elegant/ugly", or "like/hate". Note, since no design is perfect, including both pros and cons of a design (i.e., its trade-offs) or alternate designs you considered is highly encouraged.

  • Status
    Reflect on the coding details of the entire project by looking at all the code.
    • Is the code generally consistent in its layout, naming conventions and descriptiveness, and style?
    • Is the code generally readable (i.e., does it do what you expect, does it require comments to understand)?
    • Are the dependencies in the code clear and easy to find or do they exist through "back channels" (e.g., global variables, order of method call, type requirements, or get methods instead of parameters)?
    • Are there implemented features that are particularly easy or hard to extend?
    • Review three classes in the program you did not write or refactor in detail:
      • What is interesting about this code (why did you choose it? is it especially good or bad?)
      • What suggestions would you give the code's author about the code (i.e., improving it or making it easier to understand or extend)? Or what you learned about design by reading the code?
      • What would be required to make the code reusable in a completely different project (like your CellSociety or SLogo)?
  • Design
    Reflect on how the entire program is currently organized.
    • Describe the overall design of the complete program: the basic design of each sub-part and how they work together (i.e., what behavior, data, or resources each part depends on from the others).
    • As briefly as possible, describe what is needed to represent a specific game (include both code and resource files that are needed).
    • Describe how your code is designed in more detail, but still try not to reference specific data structures or actual code.
    • Justify why your overall code is designed the way it is or what issues you wrestled with if you think its design is lacking in some way. Are there any assumptions or dependencies in your code that impact the overall design of the program?
    • Describe three features from the assignment specification (at least two you did not implement) in detail:
      • What classes or resources are required to implement this feature?
      • Describe the design of this feature in detail (what parts are closed? what implementation details are encapsulated? what assumptions are made? do they limit its flexibility?)
      • How extensible is the design for this feature (is it clear how to extend the code as designed? what kind of change might be hard given this design?)
  • Alternate Designs
    Reflect on alternate designs for the project based on your analysis of the current design or project discussions.
    • How did the original design handle extensions over the course of the project's development?
    • How did the original APIs change over the course of the project?
    • Describe three design decisions discussed, either as a team of with a TA, in detail:
      • What alternate designs were proposed?
      • What are the trade-offs of the design choice (describe the pros and cons of the different designs)?
      • Which would you prefer and why (it does not have to be the one that is currently implemented)?
    • What are the two best features of the project's current design?
    • What are the two most important issues that remain in the project's current design?
Code Masterpiece

Take part of the project and refactor it to demonstrate what you now think of as good design and that you are proud to defend as well-designed according to what you have learned during the semester. It should represent a feature or component within your larger project that is small enough to be workable, but large enough to have some interesting design element. I use the term "component", rather than "class", because it may be several classes that work together, e.g., a superclass and its sub-classes or a few classes that work together. Note, if your component is specifically intended to be superclass, it must include enough detail that it is clear how to extend it and at least one example subclass to show it in "action".

In addition to the code, include a section at the end of your written analysis describing the component's purpose and why you think it is well designed (i.e., what you think it shows that you have learned so far in the course). Your code masterpiece must be something you wrote or something you refactored significantly. If you choose to refactor someone else's code, strongly justify your decision in your written analysis.

SLogo Addition

Add one of the features described here to your team's SLogo project based on the part of the project you worked on or with which you feel more comfortable working. After you have completed the code, answer the following questions in a single file, ADDITION.md, in your new branch:

  • Estimation: before looking at the old code:
    • how long do you think it will take you to complete this new feature?
    • how many files will you need to add or update? Why?
  • Review: after completing the feature:
    • how long did it take you to complete this new feature?
    • did you get it completely right on the first try?
    • how many files did you need to add or update? Why?
  • Analysis: what do you feel this exercise reveals about your project's design and documentation?
    • was it as good (or bad) as you remembered?
    • what could be improved?
    • what would it have been like if you were not familiar with the code at all?
